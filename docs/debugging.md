# Solving problems, or the art of debugging

You will inevitably bump into bugs in your code (e.g. random segmentation faults).
It is thus helpful to know what tools exist that can help you in debugging your program.

This document is by no means a thorough guideline, but rather, a introductory document to the arsenal of tools available to you.
This document therefore does not expect you to fully understand the intricacies of using these tools, but rather aims to help you develop a better understanding of what is available and provide a preview of some of the simpler things that can be done to help debug your code.
As with learning any programming skills, there are many resources online to help you on the way so long as you know what you are searching for.

Notes:
* You will likely spend at least an hour or two to go through this entire document, but doing so will likely save you many more hours down the line.
* For simplicity, it is assumed that you are in a [Unix](https://en.wikipedia.org/wiki/Unix)-based environment; you are when you use the VS Code terminal with Docker.

The procedure for solving your bugs will likely be something akin to going down the sections of this document using all the techniques one after the other, and sometimes going back up, helped by what you learned about the bug from using the previous technique.
In each section you will have to pose hypotheses about the bug, test them, and adjust, refine or change hypothesis until you know how to fix the bug, or don't progress any more in your search.

If at any point you think you are lost, or are wasting too much time on a bug, or if the tools here do not fit your problems / needs, simply searching "How to fix XXXX" (or just the error message you initially got) in a search engine will likely get you a solution or at least give you a better understanding of the problem.
You might be surprised with the results - this is how I found out about `valgrind`, one of the tools presented below.
And if that still isn't doing it, remember that TAs are here to help you.
They may not solve the issue immediately, but once they know what the issue is, they may choose to explain it to you instead of solving it for you.
The underlying goal is for you to learn - sometimes coming up with your own solution that you understand and can explain in a written report is best.

## Error messages

You should always start by reading error messages and try to understand them.
Not only will it help you locate where the bug may be, it can also help you diagnose the cause of the bug, if not straight up tell you how to fix it.

One of the first question to ask yourself with the error message is what step of `test.py` failed.
There are several steps usually well described in the log and error messages, each one pointing to a specific class of error:

* cleaning the project → optional, shouldn't fail
* producing object files from your compiler source files → you wrote invalid code
* linking together all the object files to produce your compiler executable → there is a mismatch between what one source file expects and other source file and libraries provide
* initialising coverage data → optional, shouldn't fail
* running tests one after the other → refer to the paragraph below for details
* processing coverage data → optional, shouldn't fail

Running tests is itself composed of several steps per test with distinct errors:

* generating reference assembly using gcc for RISC-V → you modified a test or added an invalid test
* generating assembly using your compiler → your compiler is crashing or stuck in an infinite loop
* assembling the assembly generated by your compiler into an object file → the assembly you generated is wrong or incomplete
* producing the executable from the object file and the driver → there is a mismatch between what the driver expects and what the assembly you generated provides (or the test is not valid, see first bullet point)
* simulating the executable → the driver/test is invalid, or the assembly produced by your compiler is not behaving correctly

## Code coverage

After reading this section, you should be able to find which part of your compiler has generated the wrong assembly you are trying to fix.
Coverage deals with the source code (`.c`) of your compiler because there are no simple ways of generating coverage data on the generated assembly (`.s`).

You can quickly pinpoint where your compiler crashes by using code coverage data after compiling a single test file.
[We provide all you need to simply produce coverage data.](./coverage.md)

1. Choose a single test file to generate coverage information as explained in `coverage.md`
2. You can start by looking at the last line executed in [`compiler.cpp`](../src/compiler.cpp) and dive into the last executed function
3. Repeat until the last executed statement is not a function (you found the crash point) or you lose track because of inheritance, loops, or recursion

If you find a section of code which is fully covered when it shouldn't be (e.g. an error detection which has been triggered), use number of times the line has been covered as a hint.
This should at least give you a better idea of where the crash is and isn't, and do so faster than inserting printing to terminal and recompiling.
It also avoids questioning your sanity because terminal output is buffered: what you print is not always displayed immediately - or at all if the program crashes at the wrong point.

## Proof-reading code

This applies to small programs, small portions of code you or your collaborators just added, and small portions of code you identified as being the source of a problem.

Writing nonsensical code, code relying on wrong assumptions, syntactically invalid code, and code riddled with typos are common sources of errors.
It happens to everyone, especially when trying to solve a problem - so don't judge yourself (or your colleagues) too harshly for that.
It also fortunately means that bugs can often be spotted (and sometimes even solved) by simply reading the code carefully while attempting to imagine what it does at a high level.

Your first reading pass should be focused on spotting syntax errors and dodgy code.
The second reading pass should attempt to make sense of the intent of the code at a high level, what non-trivial assumptions are made, and why it is doing what it does the way it does.
If not already done:

* Non-trivial assumptions should be transformed into assert statements
* Code behaviour should be captured with better names and formatting (indentations, line breaks, etc.)
* Non-trivial reasons behind the code operation should be explained with comments

Finally, the last read should be focused on carefully simulating the code step by step with the values you would have expected at the moment the bug occurred.
If you didn't find the bug yet, note down these values as they will help during the next step.
